> 原则：先把问题缩小到“最小可复现”，再解决。  
> 如果你不确定是不是 OpenClaw 的问题，先用一个空目录复现（避免把真实项目搞乱）。

## 1）安装/命令找不到

现象：
- `claw: command not found`
- `permission denied`

排查清单：

1) 确认命令是否存在
```bash
command -v claw || echo "not found"
```

2) 确认 PATH
```bash
echo "$PATH" | tr ':' '\n' | sed -n '1,120p'
```

3) 如果官方安装脚本把二进制放在用户目录，确保对应目录在 PATH 里  
（以官方文档为准：`https://docs.openclaw.ai/install/`）

## 2）权限/安全模式导致“什么都做不了”

现象：
- 工具被拒绝执行
- 需要你确认权限

建议做法：
- 第一次就把它当“新人同事”：先只读、先出计划
- 你确认计划后再逐步放开写入/执行权限

你可以把规则写进 `AGENTS.md` 或项目规则文件（所有工具通用）：
- 默认只读
- 不触碰 secrets
- 小步改动 + 可回滚

## 3）输出跑偏：写了一堆但不可用

常见原因：
- 目标不清晰（你给的是“我要一个方案”，而不是“我要一个可执行 SOP”）
- 输入不够结构化（来源链接太杂、没有约束输出格式）
- 没有验收标准（什么叫“可用”？）

解决办法：给它“验收标准”

你可以直接复制这段提示词：

```text
你要输出一份可执行 SOP（步骤编号 1..N），每一步必须可操作。
输出必须包含：
1) 目标（1 句话）
2) 前置准备（清单）
3) 步骤（编号）
4) 验收标准（如何判断完成）
5) 下一步（2 个站内链接）
不要编造数据/引用，不要输出空泛建议。
```

## 4）误改/误删怎么办（回滚策略）

强烈建议你把“回滚”变成默认流程：

- 所有输出写到单独的 `outputs/` 目录
- 对真实仓库的改动：**一次只改一个点**
- 每次完成后做一次小 commit（描述清晰）

如果你在 Git 仓库里：
```bash
git status
git diff
```

不确定就先撤销本次未提交改动（谨慎）：
```bash
git restore .
```
