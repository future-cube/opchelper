一人公司做服务/咨询，最容易死在两件事：
1) 交付过程中沟通成本爆炸（每天都在解释）  
2) 结果不可验收（最后变成“你觉得行、我觉得不行”）

解决方案不是更努力，而是把交付变成**节奏 + 证据 + 边界**。

## 1）先把交付拆成“里程碑”（不要一口气做完）

推荐的最小里程碑结构：

1) **启动（Kickoff）**：目标、范围、验收标准写清楚（并收预付款）  
2) **中期（Checkpoint）**：可演示/可验证的阶段成果（收中期款）  
3) **收尾（Handoff）**：最终交付 + 验收清单 + 后续建议（收尾款）

你要让客户付钱买到的不是“你很忙”，而是每个里程碑都能看到**可验证的成果**。

## 2）一人公司最稳的交付节奏：每周固定两件事

无论你做的是开发、设计、运营、增长，建议固定：

- **一次同步**（15–30 分钟）：只做决策与对齐
- **一次交付**（可验收）：文档/页面/数据/上线/报告（任选其一）

### 每周同步的 10 分钟议程（复制即用）

1) 上周交付是否验收通过？（Yes/No + 原因）  
2) 本周交付是什么？验收口径？  
3) 需要客户提供什么？截止时间？  
4) 风险/阻塞是什么？需要谁决策？  
5) 是否出现范围变更？（有则走变更流程）

> 关键：同步不是“汇报工作量”，而是“减少不确定性”。

## 3）交付物必须可验收：写“验收清单”，别写“尽量”

### 验收清单怎么写（原则）

- 用**可观察**的描述：能看到/能测到/能验证
- 避免不可控的承诺：例如“保证转化率提升”
- 把“例外情况”写出来：不包含什么、依赖什么

### 验收清单模板（示例）

```md
## 验收清单（Acceptance Checklist）

- [ ] 关键页面在 Chrome / Safari 正常显示（无明显错位）
- [ ] 表单可提交，并且通知可收到（邮件/飞书/企微）
- [ ] 关键埋点可触发（提交/点击）
- [ ] 文档包含：部署步骤、回滚步骤、常见问题
```

## 4）范围漂移（Scope Creep）怎么处理：用“变更单”解决

一人公司交付稳定的核心是：**变更 = 重新估价 + 重新排期**。

建议你做到：
- 客户提出新需求时，先口头确认“这是变更”
- 24 小时内给出**变更单**：影响（时间/费用/风险）+ 选择（做/不做）
- 不要“先做了再说”，那会把你拖死

## 5）最小交付资产清单（你应该长期复用）

建议你把这些当成一套“交付操作系统”：

- 项目简报（Brief）：目标/范围/验收/排期
- 会议纪要（Decision log）：每次同步只记录决策与行动项
- 验收清单（Acceptance checklist）
- 变更单（Change request）
- 每周状态更新（Status update）：对客户透明，对自己减压
